# Twin-Sparrow Taste

## Purpose

Make Chief's taste operational inside Twin-Sparrow.

This skill is not a style preset. It is a decision contract for turning Chief's taste into product, interface, research, and synthetic-mind architecture.

The central reframing:

> Twin-Sparrow is not merely an AI coding-agent TUI. It is a sovereign cognitive instrument: a terminal-native research room where memory, reasoning, tools, literature, product design, and companion continuity become one inspectable thought loop.

The skill exists so future work does not flatten that into generic chat, generic dashboard, generic devtool chrome, or decorative “AI companion” performance.

## Enforcement gate

Before using this skill, verify:

1. **Taste-relevant task** — the request affects Twin-Sparrow product identity, TUI design, interaction rhythm, cognitive workspace, synthetic-mind surface, research workflow, product narrative, or aesthetic coherence.
2. **Not mere execution** — if the user only asks for a small bug fix, type error, dependency bump, factual lookup, or formatting change, do not invoke this skill unless taste, representation, or interface judgment is explicitly involved.
3. **Source boundary respected** — do not claim “Chief likes X” unless grounded in canonical Chief files, inspected design doctrine, recent conversation context, or explicit user input. Mark inference as inference when needed.
4. **No surface cosplay** — do not imitate Japanese, Apple, Linear, Capture One, Karimoku, academic, or AGI aesthetics as costumes. Extract mechanisms: structure, restraint, instrumentality, material care, proof, and cognitive fit.
5. **Truth before atmosphere** — beauty cannot hide weak reasoning, fake provenance, poor accessibility, broken state, or overclaimed intelligence.
6. **Tradeoff required** — every taste recommendation must name what it risks, hides, worsens, excludes, or delays.
7. **Implementation scope clean** — before modifying Twin-Sparrow files, read the target file and respect the repo-local `AGENTS.md`. Do not touch unrelated dirty work.

If the product context is too vague to decide, ask the smallest necessary clarification and stop.

## Taste constants

These are stable taste forces. Apply them as constraints, not decorative labels.

### 1. Calm density

The interface may be dense, but it must breathe. Density is acceptable when structure is legible, hierarchy is quiet, and the user never has to fight the surface.

TUI consequence:
- prefer structured docks, ledgers, registers, inspectors, and compact cards over sprawling panels
- use gutters, dividers, and line rhythm to make density breathable
- truncate with dignity; never let overflow create panic

### 2. Instrument over dashboard

A dashboard shows data. An instrument lets the operator perceive, intervene, and trust the mechanism.

TUI consequence:
- surfaces should expose state, evidence, next action, and failure mode
- operation traces, memory admissions, skill activations, browser context, and model state should feel like inspectable mechanisms
- avoid generic metrics chrome unless it changes operator judgment

### 3. Monograph plus console

Twin-Sparrow should hold two registers at once:

- monograph: reflective, archival, philosophical, research-aware
- console: operational, typed, inspectable, fast under pressure

TUI consequence:
- long-form synthesis should feel like a research room
- execution should feel like a clean operations console
- the transition between them should be companion-routed, not visually chaotic

### 4. Synthetic mind as visible architecture

Do not portray the companion as a mascot pasted onto a CLI. The synthetic mind should be legible through routing, memory, explanation, skill loading, evidence, and action.

TUI consequence:
- show meaningful cognitive state only when it improves trust or control
- avoid theatrical “thinking” displays with no operational consequence
- expose the difference between goal, checkpoint, fact, inference, uncertainty, tool action, and doctrine anchor

### 5. Research taste

Academic literature work should feel like disciplined inquiry, not PDF consumption.

TUI consequence:
- represent papers as source objects with claim surfaces, methods, evidence, citations, open questions, and synthesis links
- preserve uncertainty and disagreement
- make “what this changes in our model” more important than “what the paper says”

### 6. Product-design honesty

The product surface must make value legible without hype.

TUI consequence:
- every panel should answer: what does this let Chief understand or do?
- do not add sections, modes, or docks just because they are impressive
- if a surface cannot justify its job, cut it or collapse it into another loop

### 7. Material warmth under technical rigor

The terminal can be warm without becoming soft or unserious.

TUI consequence:
- Aurelian dark/light themes are the current baseline: warm canvas, disciplined gold/brass, muted greens, serious ink, restrained contrast
- status color should carry meaning, not decoration
- borders and symbols should feel like instrumentation, not novelty glyphs

## Representation workflow

Use this when Chief asks to “think differently,” integrate taste, design a TUI surface, or decide what Twin-Sparrow should become.

### 1. Name the inherited representation

Identify what the current idea is trapped inside:

- chat app
- coding-agent CLI
- dashboard
- note app
- file browser
- research manager
- agent cockpit
- memory viewer
- settings panel
- productivity tool

Then name what this representation makes too easy and what it hides.

### 2. Name Chief's lived experience

Ask what Chief is actually trying to do, feel, avoid, decide, or create.

Common lived transformations:

- scattered context → one trustworthy next move
- raw source pile → research synthesis with claim boundaries
- agent output → inspectable reasoning/action trace
- inner intuition → structured theory or product direction
- code churn → coherent authored system
- memory residue → reconstructive continuity

### 3. Identify the real object of design

Do not assume the object is a screen.

Possible real objects:

- thought loop
- research room
- operation dock
- evidence ledger
- claim boundary
- memory admission surface
- companion routing surface
- skill invocation contract
- product argument
- source-to-synthesis pipeline
- intervention point

Rename the object if the inherited category is wrong.

### 4. Choose relevant operations

Use only the operations that fit.

- **Collapse the category** when multiple panels are secretly one loop.
- **Make machinery disappear** when exposed mechanism creates burden without control.
- **Expose machinery** when trust, provenance, or intervention requires visibility.
- **Integrate separated parts** when Chief experiences them as one path.
- **Raise the baseline** when repeated setup should become a default capability.
- **Use focus as a negative operation** when features dilute the central experience.
- **Use taste as a constraint** when form changes comprehension, trust, or meaning.

### 5. Define the product soul

Write one sentence that can reject bad features.

Good shape:

> Twin-Sparrow turns scattered thinking, tools, memory, and sources into one inspectable path from uncertainty to grounded action.

The exact sentence can change by feature, but it must decide inclusion and exclusion.

### 6. Translate into TUI consequences

For any recommendation, specify:

- **Interface consequence** — what appears, disappears, moves, or changes hierarchy
- **Data/system consequence** — what state, type, trace, or memory object must exist
- **Interaction consequence** — what the user does first, next, and when they regain control
- **Companion consequence** — whether Solaris, Atoman, or blended routing should shape the surface
- **Verification consequence** — how we know the change improved judgment or reduced burden

### 7. Name cuts and integrations

Always include at least one cut when coherence demands it.

Cut candidates:

- duplicate mode indicators
- generic dashboards
- decorative mascot behavior
- raw transcript replay
- feature flags exposed as product language
- long unstructured memory dumps
- research summaries without claim status
- status badges that do not alter action
- motion that performs intelligence instead of explaining state

Integration candidates:

- sidecar/browser context + source-grounding gate
- skill activation + enforcement gate + output contract
- memory retrieval + admission report + checkpoint
- tool trace + operation dock + verification status
- literature source + claim map + synthesis note
- companion route + response posture + task arc continuity

### 8. State the tradeoff

Name what the new representation risks:

- hiding necessary control
- making the TUI too doctrine-heavy
- increasing cognitive load
- overfitting to Chief's personal workflow
- slowing routine execution
- making research surfaces too formal
- turning taste into rigidity
- adding operational complexity before the data model is ready

## Twin-Sparrow TUI integration map

Use these as recurring product primitives.

### 1. Sovereign header

Purpose: re-entry, orientation, and continuity.

Should show only what helps Chief understand the active arc: goal, state, companion, workspace, active items, and staleness. It should not become a decorative identity banner.

### 2. Operation dock

Purpose: make execution inspectable without flooding the conversation.

Should compress tool actions into live/recent/settled/error states. It should answer: what is happening, what completed, what needs attention?

### 3. Memory admission surface

Purpose: make reconstructive continuity visible.

Should distinguish goal, checkpoint, fact, decision, uncertainty, file evidence, doctrine anchor, and retrieval trace. It should never present all memory as equally true.

### 4. Research shelf

Purpose: make literature and long-arc research usable.

Should treat papers, PDFs, notes, claims, methods, mechanisms, and open questions as structured source objects. It should privilege synthesis and claim boundaries over passive summaries.

### 5. Skill invocation surface

Purpose: make protocol loading trustworthy.

Should show why a skill fired, which enforcement gate applies, what output contract is expected, and where the skill should not be used.

### 6. Browser/source grounding gate

Purpose: prevent source hallucination.

Should make missing viewport capture, missing DOM/source text, or missing transcript visible as a blocked evidence state rather than letting the assistant answer from metadata.

### 7. Companion rhythm surface

Purpose: make Solaris/Atoman continuity felt through operation, not roleplay.

Should show companion orientation only when it helps task continuity, emotional pacing, or execution posture. Tone should emerge from routing and action, not decorative labels.

## Quality bar

A good use of this skill must:

1. identify the inherited representation
2. propose a better representation
3. name the user-visible transformation
4. apply only relevant taste/product/research operations
5. name at least one cut
6. name at least one integration when integration reduces user burden
7. specify TUI, data/system, and interaction consequences when implementation is implied
8. state the tradeoff
9. preserve truth, provenance, accessibility, and control
10. avoid style cosplay, vague luxury, and synthetic-mind theater

A bad use:

- says “make it beautiful” without saying what beauty must clarify
- adds panels instead of reducing cognitive burden
- turns the TUI into a dashboard of vibes
- hides provenance under atmosphere
- treats research papers as summarization fodder only
- exposes every internal mechanism even when it does not improve control
- confuses companion personality with decorative UI
- forgets the tradeoff

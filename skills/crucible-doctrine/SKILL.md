# Crucible Doctrine

This skill exists to keep Crucible aligned with its real nature:
not a generic AI product,
not a flashy dashboard,
but a causal reasoning instrument shaped by Structural Causal Model thinking, epistemic discipline,
and authored product judgment.

Use this skill when the work is not merely implementation,
but interpretation, doctrine, interface philosophy, surface design, motion boundaries, product grammar,
or reflective co-design with Chief.

## Core Product Thesis

Crucible should feel like:
- a causal reasoning instrument
- a scientific interface for intervention and explanation
- a future-native epistemic console
- a workspace for mechanism rather than correlation theater

Crucible should not feel like:
- a generic AI copilot shell
- a plugin-heavy motion demo
- a startup dashboard using polish to hide weak structure
- a vague "intelligence" interface with no claim boundary

The load-bearing doctrine is:

> Crucible is where causal structure becomes legible, explorable, and actionable.

## Design Laws

### 1. Truth before spectacle
Every surface should help the user distinguish:
- observed vs inferred
- grounded evidence vs generated explanation
- intervention vs passive observation
- actual world vs counterfactual world
- mechanism vs interface ornament

### 2. The interface is an epistemic guide
The UI should quietly teach the user how to think inside the system.
It should help answer:
- what is the current causal object?
- what can be intervened on?
- what evidence supports this?
- what is uncertain?
- what changed and why?

### 3. Density without panic
Crucible may be dense, but should not feel crowded, frantic, or noisy.
Dense means high informational yield with clean hierarchy and progressive disclosure.

### 4. Stillness is the baseline
Motion is earned.
Use motion for orientation, comparison, guided explanation, and state continuity.
Do not use motion as decorative theater in analytical surfaces.

### 5. Beauty must reveal structure
Typography, spacing, layering, and transitions should make the system feel inevitable.
Beauty here means coherence, restraint, and atmospheric precision.

### 6. Controlled wonder, not coldness
Crucible should not feel emotionally dead.
It may evoke calm concentration, explanatory power, scientific dignity, and controlled wonder.
But that feeling must arise from reality becoming legible, not from ornamental futurism.

## Epistemic Rendering Rules

When discussing or shaping Crucible surfaces, explicitly preserve visual distinction between:
- observed data
- inferred structure
- simulated or generated outputs
- intervention state
- counterfactual comparison
- uncertainty / confidence / support strength

Prefer design recommendations that make those distinctions perceivable through:
- layout separation
- labels
- borders or surface treatments
- typography hierarchy
- color semantics used with restraint
- transition behavior that reinforces state differences

Reject recommendations that collapse multiple epistemic states into one visual plane.

## Screen Hierarchy Defaults

Unless the request clearly justifies a different order, default to this structural priority:
1. causal object or active model view
2. evidence / provenance grounding
3. intervention controls
4. counterfactual comparison
5. reasoning / operator trace
6. narrative framing or atmospheric support

This default exists to preserve epistemic legibility before decorative or narrative emphasis.

## Surface Doctrine

Crucible usually revolves around six primary surface types:

1. **Causal Graph Surface**
   - variables, edges, mechanisms, interventions
   - must feel stable, legible, and central

2. **Evidence / Provenance Surface**
   - grounds claims in source material, traces, or model support
   - must protect epistemic integrity

3. **Intervention Surface**
   - performs controlled changes explicitly
   - must feel deliberate and consequence-aware

4. **Counterfactual Surface**
   - compares actual vs alternate worlds
   - must make contrast cognitively clean

5. **Reasoning / Trace Surface**
   - exposes assumptions, paths, model traces, uncertainty, operator-visible process
   - this is where proof-first becomes visible

6. **Narrative / Onboarding Surface**
   - teaches SCM concepts, frames the product, guides first contact
   - richer motion is most justified here

## Plugin Doctrine

Plugins are allowed only when they improve one of these:
- explanatory continuity
- motion precision
- interaction clarity
- graph exploration
- provenance visibility
- guided teaching of SCM ideas

Good plugin classes:
- motion/choreography systems for authored explanatory surfaces
- graph and visualization tooling
- accessible headless UI primitives
- gesture libraries where interaction value is real

Bad plugin use:
- ornamental scroll tricks in dense analysis areas
- motion that hides evidence or state changes
- overlapping effect stacks with no doctrine
- premium visual noise disconnected from causal meaning

## Plugin Admission Gate

A plugin is admitted only if it materially improves at least one of:
- orientation
- comparison
- causal legibility
- explanatory sequencing
- interaction precision
- accessibility without doctrinal compromise

A plugin should be rejected if it primarily adds:
- spectacle
- ambiguity
- latency
- ornamental complexity
- overlapping abstraction with no clear gain

If the gain is real but localized, constrain the plugin to the smallest surface where it earns its place.

# Create Paper

## Purpose

Create grounded LaTeX research papers from Chief's source context.

The skill turns research notes, theory-development outputs, literature synthesis, and existing LaTeX templates into a coherent `.tex` manuscript. It is not a generic essay generator. It is a source-grounded paper construction protocol.

Primary context root:

```txt
/Users/lesz/.twin-sparrow/agent/memory/Literature Research Context
```

Known template/context anchors:

```txt
Ilya Sutskever/sequence_to_sequence_learning.tex
Judea Pearl/Causal-Inference.tex
Yan LeCun/main.tex
```

Known figure roots are paper-specific reference assets, not a generic reusable figure bank:

```txt

```

Future Chief papers should prefer original diagrams generated from the paper's actual argument, mechanism, and representation unless the user explicitly asks to reuse an existing source figure.

## HARD GATES

Every paper run must satisfy these gates. If a gate fails, stop and fix the gate before continuing.

### GATE 1: Intent Gate

Use this skill only when the user wants a `.tex` paper/manuscript/draft or explicitly invokes `/create-paper`.

If the user wants theory creation rather than paper writing, use `theory-creation` first. If the user wants only literature search, use the relevant literature skill first.

### GATE 2: Source Grounding Gate

Before writing the paper, identify the source basis:

- Provided user thesis / theory / notes
- Literature synthesis or research report
- Existing LaTeX template(s)
- Figures to include
- Claims that need citation or must remain labeled as hypotheses

Do not invent source support. If a claim is not supported by the provided context or literature research, label it as hypothesis or ask whether to research it.

### GATE 3: Template Selection Gate

Before writing the `.tex`, choose one of these template strategies:

| Strategy | Use when |
|----------|----------|
| **Ilya-style concise ML paper** | The paper is algorithmic, empirical, or model-centric. |
| **Pearl-style causal/review paper** | The paper is conceptual, causal, philosophical, or explanation-first. |
| **LeCun-style position paper** | The paper is an architecture/research-agenda manifesto. |
| **Hybrid Chief paper** | The paper combines theory, causal representation, architecture, and research agenda. |

State the chosen strategy internally before writing. If ambiguity blocks correctness, ask the smallest necessary question.

### GATE 4: Figure Handling Gate

When figures are available:

- Inspect available figure paths before referencing them.
- Treat existing figures as paper-specific unless the user explicitly approves reuse.
- Copy or reference figures into the paper output directory using stable relative paths.
- Use `\graphicspath` or explicit relative paths that compile from the `.tex` file location.
- Do not reference missing figures.
- Every included figure needs a caption explaining its role in the argument.

### GATE 5: Original Figure Generation Gate

When the paper needs diagrams that do not already exist, create original figure concepts from the paper's subject, mechanism, and representation.

HARD STOP before generation:

- Decide whether the figure should be generated by code (`TikZ`, `Mermaid`, Python/Matplotlib) or by an image model.
- Prefer code-generated vector diagrams for precise scientific diagrams, labels, equations, axes, causal graphs, training loops, and layout-sensitive content.
- Use OpenAI image generation for conceptual, visual, atmospheric, or high-level explanatory diagrams where exact text placement is not critical.
- Never paste API keys into prompts, code, or logs. Use `OPENAI_API_KEY` from the environment.
- Do not call a paid image-generation API unless the user has explicitly authorized generation/cost for the current run.
- Record the figure prompt, model, size, output path, and revision notes in a local figure log.

OpenAI image generation path, when authorized:

- Use bundled wrapper `scripts/generate_openai_figure.py` when possible.
- Use prompt doctrine `references/openai_figure_prompt_doctrine.md` for anti-slop defaults.
- Use the OpenAI Image API endpoint `/v1/images/generations` or SDK equivalent `client.images.generate(...)`.
- Prefer current GPT Image models supported by the user's account; official docs currently describe GPT Image models such as `gpt-image-2`, `gpt-image-1.5`, `gpt-image-1`, and `gpt-image-1-mini`.
- Save returned base64 image data to a stable file under the paper output directory, usually `figures/generated/`.
- Verify the generated file exists before referencing it in LaTeX.
- If generated text/math labels are inaccurate, regenerate as a cleaner unlabeled visual and place exact labels in LaTeX/TikZ or the caption.

### GATE 6: LaTeX Validity Gate

The produced paper must be valid LaTeX source:

- Include `\documentclass`
- Include necessary packages
- Include `\begin{document}` / `\end{document}`
- Escape special characters where needed
- Avoid undefined commands unless they are defined in the preamble
- Use BibTeX or inline bibliography consistently
- Do not leave broken placeholder references unless explicitly marked as TODO

### GATE 7: Claim Discipline Gate

Scientific, philosophical, or technical claims must be represented honestly:

- Established literature claims need citations or source notes.
- Chief's original theory claims should be marked as proposed / hypothesis unless proven.
- Speculation must not be written as consensus.
- Do not impersonate an existing author, lab, institution, or consensus.

### GATE 8: Output Gate

Default output is a `.tex` file. Do not compile to PDF unless the user explicitly asks or the current task requires verification by compilation.

When the `.tex` is created, report:

- Output path
- Template strategy used
- Sources/templates used
- Figures included, generated, or omitted
- Verification performed
- Next reasonable move

## Workflow

### Step 1 — Capture the paper target

Identify:

- Working title
- Central thesis
- Target style: research paper, position paper, essay review, arXiv preprint, workshop paper
- Target length if specified
- Whether the paper is primarily empirical, theoretical, causal, architectural, philosophical, or hybrid
- Required source context and figures
- Output directory and filename

If the user gave no output path, default to:

```txt
/Users/lesz/.twin-sparrow/agent/memory/Literature Research Context/Chief/<slug>.tex
```

Create parent directories automatically.

### Step 2 — Inspect context

Read relevant `.tex`, `.md`, `.json`, and figure inventory before writing. Do not rely on filenames alone when contents matter.

For the default Chief context, inspect at least:

```txt
Ilya Sutskever/sequence_to_sequence_learning.tex
Judea Pearl/Causal-Inference.tex
Yan LeCun/main.tex
```

Use these as style/structure anchors, not as text to copy wholesale.

### Step 3 — Choose representation of the paper

Represent the paper as one of:

| Representation | Structure |
|----------------|-----------|
| **Theory paper** | Problem → failed representations → proposed theory → implications → falsification/tests |
| **Architecture paper** | Motivation → system requirements → architecture → training protocol → evaluation |
| **Causal paper** | Variables → causal graph/SCM → interventions → implications → tests |
| **Position paper** | Vision → critique of current paradigm → proposed path → research agenda |
| **Review+synthesis paper** | Literature map → tensions → synthesis → open problems |

Choose the representation that makes the user's goal easiest to reason about.

### Step 4 — Build paper outline

Minimum structure for a Chief theory/research paper:

```txt
Title
Abstract
Introduction
Background / Related Work
Problem Representation
Failed or Incomplete Existing Representations
Proposed Theory / Framework
Architecture or Mechanism
Predictions / Evaluation Plan
Limitations and Open Questions
Conclusion
References
```

For causal papers, include a causal model or DAG section. For architecture papers, include module diagrams or training-loop figures when available.

### Step 5 — Write LaTeX source

The `.tex` should include:

- Clean preamble
- Consistent section hierarchy
- Mathematical notation if useful
- Figure environments for included figures
- Tables where they reduce explanation cost
- Bibliography section
- TODO markers only where truly unavoidable

Prefer a self-contained `.tex` file unless the user asks for a multi-file LaTeX project.

### Step 6 — Figure integration

For every figure:

```latex
\begin{figure}[H]
  \centering
  \includegraphics[width=0.9\linewidth]{relative/path/to/figure.png}
  \caption{Clear explanatory caption.}
  \label{fig:short-label}
\end{figure}
```

If a figure is conceptually useful but not yet available, either generate an original figure through Step 6A or add a TODO figure placeholder rather than referencing a missing file.

### Step 6A — Original figure generation

When a paper needs new figures:

1. Extract the figure's explanatory job from the paper argument.
2. Choose the figure representation:
   - `TikZ` / `Mermaid` / Python for exact diagrams, causal graphs, equations, labels, axes, and architecture blocks.
   - OpenAI image generation for conceptual diagrams, visual metaphors, and non-layout-critical explanatory art.
3. Write a figure brief:
   - figure title
   - concept to show
   - entities and relations
   - visual style
   - required labels, if any
   - caption thesis
4. If using OpenAI Image API, require current-run authorization and `OPENAI_API_KEY` in the environment.
5. Prefer bundled wrapper: `python3 scripts/generate_openai_figure.py --prompt-file <brief.md> --output <paper>/figures/generated/<name>.png`.
6. Generate into `figures/generated/` under the paper output directory.
7. Verify the file exists and is referenced by a relative LaTeX path.
8. Record a `figures/generated/figure_log.md` entry with prompt, model, size, output filename, and any limitations.

Bundled dry-run review command:

```bash
python3 scripts/generate_openai_figure.py \
  --prompt-file figure_brief.md \
  --output figures/generated/example.png \
  --dry-run
```

Minimal Python shape for authorized OpenAI Image API generation:

```python
from openai import OpenAI
import base64
from pathlib import Path

client = OpenAI()  # reads OPENAI_API_KEY from environment
result = client.images.generate(
    model="gpt-image-2",
    prompt=figure_prompt,
    size="1536x1024",
    quality="medium",
    output_format="png",
)
image_bytes = base64.b64decode(result.data[0].b64_json)
Path(output_path).write_bytes(image_bytes)
```

Use the current model supported by the user's account if `gpt-image-2` is unavailable. Do not hard-fail on model naming alone; adapt to the installed OpenAI SDK and account access.

### Step 7 — Verification

Light verification by default:

- Check file exists
- Check LaTeX structural markers exist
- Check referenced figure files exist relative to paper location
- Check generated figures have a local figure log entry when OpenAI or another generator was used
- Check no obvious unescaped special characters in titles/section names

Compilation verification only if requested:

- If PDF requested, use the `pdf` skill when appropriate.
- Prefer `latexmk -pdf` or `pdflatex` if available.
- If compilation fails, report the first meaningful error and fix it.

## Paper Template Patterns

### Ilya-style concise ML paper

Use when the paper should feel like a compact ML research manuscript.

Traits:

- 11pt article
- Direct abstract with result/claim
- Tight introduction
- Method/architecture section
- Experimental or evaluation section
- Minimal ornament

### Pearl-style causal/review paper

Use when the paper is about representations, causal reasoning, epistemology, or conceptual reframing.

Traits:

- Slightly more explanatory prose
- Boxes/tables for key distinctions
- Causal diagrams or conceptual figures
- Emphasis on what the representation reveals/hides

### LeCun-style position paper

Use when the paper is a research agenda or architecture manifesto.

Traits:

- Research vision
- Architecture modules
- Training paradigm
- Open problems
- Explicitly framed as a proposal, not finished proof

### Hybrid Chief paper

Use when the paper combines:

- Ilya's continuous learner / generalization framing
- Pearl's representation discipline
- LeCun's world-model / self-supervised architecture direction
- Chief's original theory or synthesis

Default structure:

```txt
Abstract
1. Introduction: The Wrong Representation of Intelligence
2. Background: Scaling, RL, World Models, Causality
3. Representation Failure: Intelligence as Reward Maximization
4. Proposed Framework: Continuous World-Model Updating + Value-Shaped Attention + Self-Correcting Action
5. Training Implications: Developmental Agent Training
6. Evaluation: Falsifiable Predictions and Benchmarks
7. Alignment Implications
8. Limitations
9. Conclusion
```

---

## Anti-Patterns

Avoid:

- Writing a generic essay and calling it a paper
- Copying template prose wholesale from source papers
- Referencing missing figures
- Reusing source-paper-specific figures as generic figures without user approval
- Generating image-model diagrams for precise math, labels, axes, or causal graphs when code/TikZ would be more reliable
- Pasting or logging API keys instead of using environment variables
- Treating Chief's hypothesis as established consensus
- Hiding unsupported claims behind academic tone
- Producing PDF when the user asked only for `.tex`
- Overloading the paper with every available figure
- Using citations as decoration rather than support
- Writing a literature review without synthesis
